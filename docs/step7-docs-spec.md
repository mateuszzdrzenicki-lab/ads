# Step 7 — Documentation layer spec

> Brief for Claude Code. Author the docs layer for the DS repo.
> Read `docs/DS-Decision-Log.md` first. Do not change component code or token values.

---

## 1. Content model — LOCKED

Three genres. Every piece of content belongs to exactly one. If you're unsure where something goes, ask rather than guess.

| Genre | Question it answers | Lives in | Register |
|---|---|---|---|
| **Decisions** | *Why* did we choose this? | `docs/DS-Decision-Log.md` | narrative, historical, versioned |
| **Guidance** | *When / how* should I use this well? | `docs/components/*`, `docs/patterns/*` | prose, human-facing |
| **Rules** | What must an agent always/never do? | `AGENTS.md`, `docs/constraints.md`, + rules blocks | terse, imperative, machine-checkable |

**Anti-duplication rule:** guidance and rules are the same knowledge in two registers. Never write the same fact in two files. Component docs carry **both registers in one file** (prose section + rules block). `AGENTS.md` and `constraints.md` carry rules **only** — no rationale, no prose.

**Never restate token values in prose.** Values are generated into `globals.css` from Figma. Docs describe *intent*; the CSS is the value. A hex in a doc is a second source of truth and will go stale.

## 3. Per-file spec

### 3.1 `AGENTS.md`
Currently only holds an autogenerated Next.js block. **Keep that block intact** (inside its `BEGIN/END` markers) and add a design-system section around it.

Must contain, in this order:
1. **One paragraph:** what this repo is (design system on shadcn/ui, Figma is token SoT, tokens generated not hand-written).
2. **Read-first routing:** which file to open for which question — a 4-row table pointing at `constraints.md`, `tokens.md`, `components/`, `DS-Decision-Log.md`.
3. **Hard rules** (terse, imperative, no rationale):
   - Never hardcode a color, radius, or spacing value.
   - Never use Tailwind palette utilities (`bg-gray-100`, `text-red-500`, …). Only semantic token utilities.
   - Never edit the generated block in `globals.css`; run `npm run tokens:build`.
   - Never invent a component that isn't in `docs/components/`.
   - Never add a token. Tokens originate in Figma (§1 of the log).
   - Before building anything, read `docs/constraints.md`.
4. **Pointer** to the degradation rule (§3.2), stated in one line.

Keep the whole DS section under ~60 lines. Density over completeness.

### 3.2 `docs/constraints.md`
The guardrail file. The single highest-value doc for AI consumption — its job is to bound the system, not describe it.

Must contain:
- **What exists right now.** Components: Button only. Variants: `primary`, `outline`. Size: `md` only. States: `default` / `hover` / `focus-visible` / `disabled`. Semantic tokens: the 12 in §6b.
- **What explicitly does NOT exist yet** — each with a one-line "so don't…":
  - No Icon atom (decorators take arbitrary children; see `components/button.md`)
  - No dark mode (deleted, not deferred — §9)
  - No status colors (`success` / `warning` / `destructive` / `info`) — primitives exist for some hues but **no semantics**; unaliased primitives are not usable
  - No spacing/sizing tokens — Tailwind utilities only, path (a) of §4
  - No typography tokens in use — `text/size/*` exist but are not consumed
  - No components other than Button
- **The degradation rule — LOCKED: stop-and-report.**
  Write it as an explicit agent instruction, roughly:
  > If a task requires a component, variant, size, state, or token that is not listed above, **stop**. Do not stub it, do not approximate it with raw Tailwind, do not invent a token. Report exactly what is missing and what would need to be added to the design system, then wait for a decision.
- **Why stop-and-report** — one sentence only: silent approximation produces output that looks correct and violates the system, which is worse than no output.

### 3.3 `docs/tokens.md`
Semantic **intent**, not values.

- One row per semantic token: token → the code utility(-ies) it drives → what it's *for* → when NOT to use it.
- State the two-tier model in two lines: primitives have no consumers; only semantics are usable.
- State the generation path: Figma → JSON export → `scripts/build-tokens.js` → `globals.css` generated block → `@theme inline` → Tailwind utility.
- Call out the non-stock `--primary-hover` explicitly, since agents trained on stock shadcn will reach for `hover:bg-primary/90`. The rule: **use `hover:bg-primary-hover`**.
- Accessibility rules that belong to tokens:
  - Never apply an opacity modifier to a contrast-carrying value (rings, borders, text). Reason: it silently drops contrast below WCAG thresholds. `disabled:opacity-50` is the one exemption (disabled controls are exempt from 1.4.3 / 1.4.11).
  - Known accepted deviation: `--border` does not meet 3:1 against `--background`. Recorded, deferred with the palette retune — do not "fix" it locally.

### 3.4 `docs/components/_TEMPLATE.md`
Extract the structure `button.md` already uses, so every future component doc is uniform. Sections, in order:

1. **Title + source links** — component path, Decision Log section refs
2. **Purpose** — one paragraph, human register: what this is for
3. **Props table** — prop / values / default
4. **API contracts** — anything non-obvious a consumer must do (e.g. Button's `data-icon` requirement)
5. **Guidance — Do / Don't** — prose, human register. Usage, composition, when to pick which variant
6. **Accessibility** — required roles/labels, focus behaviour, contrast notes, what's already handled by the component vs the consumer's job
7. **Rules block** — terse, imperative, one rule per line, agent-parseable. This is the machine register of §5/§6. Fence it and label it clearly so it's greppable.
8. **Figma parity** — the Figma component's props and how they map to code

### 3.5 `docs/components/button.md`
Extend the existing file to match the template. Do **not** rewrite what's there — the props table, decorator contract, and states sections are correct.

Add: Purpose, Guidance (Do/Don't), Accessibility, Rules block, Figma parity.

Guidance content to include (this is genuinely missing knowledge, so draft it and flag anything you're unsure of rather than inventing confidently):
- When `primary` vs `outline`
- One primary action per view/section
- Label writing: verb-led, sentence case, no trailing punctuation
- Decorators are supplementary — never the sole carrier of meaning (an icon-only button has no accessible name)
- Don't nest interactive elements inside a Button

Accessibility content:
- Renders a native `<button>` via Base UI → keyboard and role handled
- Focus ring is solid, not alpha — do not add opacity
- Disabled uses `disabled:opacity-50` and `pointer-events-none`; note that disabled controls are not keyboard-focusable, so don't rely on them to communicate state
- Any icon-only usage requires an accessible name — but note that **no icon-only variant exists**, so this is a constraint, not a recipe

## 5. Acceptance check

After authoring, verify by reading the docs as if you had no other context:

1. Could an agent that read only `AGENTS.md` + `constraints.md` correctly refuse a request for a `Card` component?
2. Could it build a correct primary Button with a start decorator using only `components/button.md`?
3. Is there any hex value, token value, or rationale duplicated between two files? If yes, remove one.
4. Does any doc describe something that doesn't exist in the repo?

Report the answers before committing.
