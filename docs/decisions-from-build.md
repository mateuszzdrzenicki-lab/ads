## 1. Style Dictionary → hand-rolled script (relates to §7.2 — already folded in)

**Divergence:** the log anticipated a Style Dictionary transform (§6/§7.2). Built instead
as a zero-dependency Node script, [`scripts/build-tokens.js`](../scripts/build-tokens.js).

**Reasoning:** the preprocessing §6d requires — resolving `$extensions.aliasData` instead
of trusting `$value`, unwrapping the color object to `.hex`, appending `px` to unitless
numbers — is fully custom regardless of tooling. Style Dictionary would only be handling
the parts we're *not* customizing (file I/O, basic parsing), while adding a dependency and
its own plugin/config API on top of the custom preprocessor we'd write anyway — working
against §8's "low coding experience, hand-held" constraint. A ~120-line script is something
that can be read top to bottom.

**Status:** already reflected in §7.2 (marked `LOCKED`, built & verified).

## 2. shadcn is a runtime dependency, not just copied-in source (relates to §3 — NEW, not yet in the log)

**Divergence:** §3 says *"shadcn is owned, not a live dependency... there is no upstream
auto-update fighting us."* In the current shadcn CLI generation, that's only half true:

- Component **source** (`src/components/ui/button.tsx`) is fully copied in and freely
  edited — no upstream auto-update touches it. This part holds.
- But `package.json` lists `"shadcn": "^4.14.0"` under `dependencies` (not
  `devDependencies`), and `src/app/globals.css` line 3 does
  `@import "shadcn/tailwind.css";` — an actual npm package imported at build time.

**What that import actually is** (verified by reading `node_modules/shadcn/dist/tailwind.css`,
16KB): it's a shared library of Tailwind v4 primitives that shadcn/Base UI registry
components lean on — `accordion-down`/`accordion-up` keyframes, data-state custom variants
(`data-open`, `data-closed`, `data-checked`, `data-selected`, `data-disabled`, etc.), and
utility families for scroll-fade and text-shimmer effects. **Button uses none of it** — no
data-state attributes, no scroll-fade, no shimmer. But any *future* component pulled via
`npx shadcn add <x>` (Accordion, Tabs, Select…) likely will — Accordion specifically needs
those exact keyframes.

**Why this happened:** it's a structural change in how the shadcn CLI packages itself, not
a choice made in this build. Older shadcn/ui shipped nothing like this; the current
Base UI–era CLI does.

**Recommendation (needs your call, not made here):**
- **(a)** Amend §3 to note this one shared stylesheet as an accepted, explicit exception —
  "owned" describes component source, not this shared utility layer.
- **(b)** Inline only the rules Button actually needs (none, currently) and drop the import,
  fully honoring §3's letter — at the cost of having to re-add pieces of this file by hand
  whenever a future component needs them.

I'd lean **(a)**: the import is inert for Button today, and manually re-deriving Accordion's
keyframes later is exactly the kind of "fighting the tool" §3 was written to avoid. But this
is a judgment call for you to make, not something I decided unilaterally.

## 4. Base font is hardcoded, not generated from the token pipeline (relates to §6a — clarification, not a contradiction)

§6a already marks `font/*` primitives "unused by the slice; typography lands in path (b)" —
so this isn't a contradiction. Worth recording anyway: the app's actual base font (Open Sans,
wired in `layout.tsx` via `next/font/google`) was set by hand-matching Figma's
`font/font-face` variable value, not generated by `build-tokens.js`. When path (b) tokenizes
typography, this hardcoded font loader will need to be replaced by a generated value, the
same way color/radius already are.

## 5. Dark mode was deleted, not deferred (gap, now resolved)

§4 mentions the `:root`/`.dark` pattern only to describe how Tailwind v4's token model works
generically — it never states whether dark mode itself is in scope for this project. The
repo's `.dark` block was stock shadcn boilerplate from scaffolding, never generated from
Figma tokens (there is no dark-mode collection in the Figma export — only "Light Mode"
exists), and nothing in the app ever applied the `.dark` class (confirmed via grep — no
`next-themes`, no toggle component, no other reference).

**Resolved 2026-07-23: the `.dark` block was deleted**, not commented-and-kept, because
there's no Figma collection to eventually generate it from — a "not yet generated" comment
would have been permanently true, not temporarily true. Also recorded directly in
DS-Decision-Log §9 (Parked): dark mode deleted rather than deferred-in-code; revisit trigger
is a dark-mode collection actually being authored in Figma, since Figma is the token source
of truth (§1).

## 6. Focus-ring contrast — new principle, applied repo-wide

The log never stated a contrast requirement for focus indicators. **Fixed 2026-07-23** in two
places, both changed from a 50%-opacity ring/outline color to the solid token:

- `src/components/ui/button.tsx`: `focus-visible:ring-ring/50` → `focus-visible:ring-ring`.
- `src/app/globals.css`, `@layer base { * { @apply border-border outline-ring/50; } }` →
  `outline-ring` (no `/50`). This rule is the native-outline focus fallback for *every*
  element in the app, not just Button — leaving it at 50% would have baked a sub-3:1 focus
  indicator into every future component.

**Numbers** (against the current `--ring` value, `#A05108` — note the original ask cited
1.68:1/3.01:1, computed against an earlier `--ring` value, `#C68346`, before two subsequent
token reapplies; recomputed against the live value below): 50%-opacity-over-white ≈ **2.2:1**
(fails WCAG 2.2 SC 1.4.11's 3:1 non-text-contrast threshold), solid ≈ **5.7:1** (passes with
margin). Conclusion is unchanged by the token updates — solid was correct before and after —
only the precise ratios moved.

Worth adding to the log as a locked principle: **focus indicators use a solid color, not a
reduced-opacity one.**
